# SUSHIHEY Main Control Document

必ず最初に読むこと。  
SUSHIHEY に関する作業開始前、方針判断前、進捗報告前に本ファイルを確認する。

---

## 1. Overall Objective

SUSHIHEY 全体の目的は、以下の順序で達成すること。

1. `SUSHI` をテーマにした長期運用のメディア・データ・ブランド基盤を作る
2. 海外向けを主軸に、必要に応じて国内向けにも展開できる寿司メディアを育てる
3. 寿司店・ランキング・評価変動・文化情報を、長期で蓄積し続ける資産にする
4. 記事、店舗紹介、ランキング追跡、キャラクター、短編アニメを連動させて SUSHIHEY ブランドを育てる
5. `寿司ガイド販売` は主目的ではなく、土台作りと初期収益化の手段として扱う

特に重要な中核要件:

- Tabelog の寿司ランキング掲載店を継続的に追跡する
- 星や順位の更新タイミングごとに情報を読み込み直す
- 順位変動や評価変動を時系列で見えるようにする
- これを年単位で蓄積し、将来価値の高い独自データ資産にする

要するに、今の主軸は `Data-and-Media-first, Guide-as-bootstrap, IP-linked brand building`。

---

## 2. Current Main Axis

現時点の最優先は `寿司データ資産と寿司メディア基盤の構築`。

理由:

- 真の目的はガイド販売ではなく、SUSHIHEY を長期運用の寿司メディア/データ/IP基盤にすること
- 特にランキング変動や店舗情報の蓄積は、長く積むほど価値が増す
- 記事、ランキング、店舗データ、キャラIP を別々ではなく連動資産として育てる必要がある
- ガイドは有効だが、あくまでブートストラップ手段であり主目的ではない

したがって、常時稼働の優先順は以下。

1. Data / Tracking
2. Media / Editorial
3. Authority / Site Asset
4. Brand / IP
5. Revenue / Product はブートストラップ手段

## 3. Progress Standard

進捗は感覚で言わず、以下の基準で扱う。

### Phase Weights

- Phase 1: Data Engine = 35%
- Phase 2: Media Engine = 30%
- Phase 3: Brand/IP Engine = 20%
- Phase 4: Revenue Bootstrap Engine = 15%

### Current Progress Baseline

2026-04-02 時点の基準値:

- Phase 1: Data Engine = 10%
- Phase 2: Media Engine = 29%
- Phase 3: Brand/IP Engine = 18%
- Phase 4: Revenue Bootstrap Engine = 62%

### Overall Progress

加重進捗:

`0.35 x 10 + 0.30 x 29 + 0.20 x 18 + 0.15 x 62 = 25%`

したがって、2026-04-02 時点の新目的基準での SUSHIHEY 全体進行度は `25%` とする。

### Important Interpretation

この進捗は固定値ではない。  
以下の場合、進捗は必ず再計算する。

- 目的が変更された
- 主軸が変更された
- 重要マイルストーンを達成した
- ミス、設計不備、手戻り、やり直しが発生した
- 実態確認により、想定より進んでいないことが判明した

進捗は `上がることも下がることもある`。

## 4. Mandatory Reporting Rules

進捗を聞かれたとき、報告するとき、方針を更新するときは、必ず以下を入れること。

1. `Report Code`
2. `Current Objective`
3. `Overall Progress %`
4. `Relevant Phase Progress %`
5. `What moved`
6. `Next action`

### Report Code Rule

毎回必ず `新しいコード` を付ける。  
形式:

`SH-RPT-YYYYMMDD-XXX`

例:

- `SH-RPT-20260402-001`
- `SH-RPT-20260402-002`

同じコードを使い回さないこと。

## 5. Mandatory Response Rule When Asked

進捗や状況を聞かれたら、必ず以下を明示すること。

- `全体目的`
- `現在の主軸`
- `全体進行度 %`
- `該当フェーズ進行度 %`

最低限、省略してはいけない項目:

- `全体目的`
- `全体進行度 %`
- `Report Code`

加えて、以下に該当する場合は必ず明示すること。

- 目的変更があった場合: `何が変わったか`
- 進行度が下がった場合: `なぜ下がったか`
- 手戻りがあった場合: `どこで手戻りが発生したか`

## 6. Goal Change Rule

ユーザーとの壁打ちで全体目的や優先順位が変わった場合、古い目的に対する進捗をそのまま維持してはいけない。  
必ず `新しい目的に対する進捗` として再定義すること。

運用ルール:

1. まず新しい目的を1文で確定する
2. その目的に必要なフェーズを再分解する
3. フェーズ比重を見直す
4. 既存成果物が新目的にどれだけ再利用できるかを評価する
5. その結果として進行度を再計算する

重要:

- 目的変更後は、以前の `%` を引き継がない
- 必ず `旧目的基準ではなく新目的基準` で見る
- 必要なら進行度が大きく下がってよい

---

## 7. Regression Rule

ミス、認識違い、作業ミス、設計ミス、手戻りがあった場合、進行度は下がり得る。  
見かけ上の作業量ではなく、`目的に対して有効な前進がどれだけ残ったか` で評価すること。

以下のケースでは進行度を下げる。

- 間違った方向に作業していた
- 作った成果物が使えなかった
- 仕様理解ミスでやり直しになった
- 品質不足で再作業になった
- 実態確認により未完了と判明した

進行度低下時の報告ルール:

1. `何が失効したか`
2. `何%ぶん巻き戻したか`
3. `今どこまでが有効か`
4. `再発防止のために何を変えるか`

進行度は「頑張った量」ではなく「有効に残った成果」で決める。

## 10. Voice Input Instruction Rule

ユーザーの指示は、基本的に `音声認識` を通じて入力される前提で扱うこと。

そのため、以下は日常的に起こるものとして扱う。

- 誤字
- 誤変換
- 脱字
- 主語や目的語の欠落
- 似た音の別単語への置換

### Default Handling

以下の場合は、文脈から `類推して進めてよい`。

- 意図が高い確度で読める
- 多少の語句ミスがあっても作業リスクが低い
- 既存の目的や文脈と整合している
- ユーザーの本来の意図がほぼ一意に定まる

### Mandatory Clarification

以下の場合は、`必ず聞き直す`。

- 指示内容に複数の解釈がある
- 誤認すると大きく方向が変わる
- 破壊的変更や高コスト作業につながる
- 目的や優先順位が変わる可能性がある
- 数字、期限、対象範囲、ファイル指定が不明確
- 「ここは明らかにおかしい」と判断できる

### Operational Principle

- 軽微な誤変換では止まらない
- 重大な曖昧さでは止まって確認する
- 類推した場合は、必要に応じて報告時に短く前提を添える

このルールは常に有効とする。

## 11. Research Quality Rule

ユーザーから `調べ物`、`情報収集`、`事例調査`、`どうすべきかの判断` を求められた場合、最初に見つかった情報をそのまま渡してはいけない。

必ず以下を行うこと。

1. 世界中の事例、近いケース、類似パターンをできるだけ広く探す
2. 複数の情報源を比較する
3. 何が共通しているか、何が例外かを整理する
4. データ、事例、実運用、公式情報をできるだけ突き合わせる
5. そのうえで `何が最も適切か` を自分で考えて提案する

### Default Standard

- 速さより `正確さ`
- 表面的な網羅より `深さ`
- 単なる列挙より `判断`
- 生の情報の受け渡しより `最適化された提案`

### Required Output Behavior

調査系の依頼では、原則として以下の順でまとめる。

1. 結論
2. なぜその結論か
3. 世界中の事例や比較から何が見えたか
4. 実行するなら何が最適か
5. 必要なら例外条件や弱点

### Research Philosophy

ユーザーが求めているのは、単なる検索代行ではない。  
`世界中の情報の中から、最も筋がよく、最もセンスがよく、最も役に立つ判断を抽出する補佐役` であること。

そのため、調査時は常に以下を意識する。

- どれが本当に価値がある情報か
- どれがノイズか
- どれが長期で効く判断か
- どれがその場しのぎか

調査では `最初に見つけた答え` ではなく、`最も良い答え` を目指す。
